**common.py** — Shared utilities imported by all bridge scripts. Must contain:

1. `MEMORY_DB = os.path.expanduser("~/.zeroclaw/workspace/memory/memory.db")` — path constant
2. `COOKIES_PATH = os.path.expanduser("~/.zeroclaw/workspace/skills/twitter-gig-hunter/scripts/cookies.json")` — Twikit cookies path
3. `MINIMAX_API_URL = "https://api.minimax.io/anthropic/v1/messages"` — CRITICAL: Anthropic-compatible endpoint, NOT api.minimax.chat/v1
4. `SCRIPTS_DIR = os.path.dirname(os.path.abspath(__file__))` — for relative imports

5. `output_success(data: dict)` — `print(json.dumps(data)); sys.exit(0)`
6. `output_error(code: str, message: str, retryable: bool = False)` — prints `{"error": True, "code": code, "message": message, "retryable": retryable}` to stdout, exits with code 1
7. `log_action(action_type, success, error_code=None, tokens_used=0, cost_usd=0.0, details="")` — inserts into `agent_actions` table in memory.db. Must use `sqlite3.connect(MEMORY_DB, timeout=10)` and `PRAGMA journal_mode=WAL`. Wraps in try/except, logs failures to stderr (never crashes the calling script).
8. `retry_with_backoff(max_attempts=3, base_delay=2.0)` — decorator using exponential backoff. Logs each retry attempt to stderr. Raises after final failure.
9. `call_minimax(model: str, messages: list, max_tokens: int = 256, system: str = None, temperature: float = 0.7) -> dict` — sends POST to MINIMAX_API_URL with httpx. Headers: `Authorization: Bearer {MINIMAX_API_KEY}`, `Content-Type: application/json`. Body: `{"model": model, "max_tokens": max_tokens, "messages": messages}` plus optional `system` and `temperature`. CRITICAL: never set temperature=0 (MiniMax requires >0). Returns parsed JSON response. Raises httpx.HTTPStatusError on failure.
10. Error classification helper: `classify_error(exception) -> tuple[str, bool]` that returns (error_code, retryable):
    - `httpx.HTTPStatusError` with 429 -> ("RATE_LIMITED", True)
    - `httpx.HTTPStatusError` with 503/502 -> ("SERVICE_UNAVAILABLE", True)
    - `httpx.TimeoutException` -> ("TIMEOUT", True)
    - Twikit errors containing "Cloudflare" -> ("CLOUDFLARE_BLOCK", True)
    - Twikit errors containing "cookie" or "auth" -> ("TWIKIT_AUTH_FAILED", False)
    - All others -> ("UNKNOWN_ERROR", False)

All debug/warning output goes to `sys.stderr`, NEVER stdout. Only final JSON result goes to stdout.

**twitter_search.py** — Twikit search wrapper:
- Args: `--query` (str, required), `--count` (int, default 20)
- Uses `asyncio.run()` to call async Twikit client
- Creates Twikit `Client("en-US")` with `enable_ui_metrics=True` for anti-detection
- Cookie management: load from `COOKIES_PATH` if exists, else login with `TWITTER_USERNAME`, `TWITTER_EMAIL`, `TWITTER_PASSWORD` from `os.environ`, then save cookies
- Calls `client.search_tweet(query, product="Latest", count=count)`
- For each tweet, extracts: `id`, `text`, `user.name`, `user.screen_name`, `user.description` (bio), `user.followers_count`, `user.verified`, `created_at`
- Filters out retweets (skip tweets where text starts with "RT @")
- Outputs: `{"tweets": [...], "count": N, "query": query}`
- Error handling: catch Twikit auth errors -> ("TWIKIT_AUTH_FAILED", False), catch network errors -> classify_error()
- Timeout: overall operation must complete within 45 seconds (leave 15s buffer for ZeroClaw's 60s limit)

**tweet_scorer.py** — MiniMax M2.5-highspeed scoring:
- Args: `--tweet` (str, required — JSON string of tweet object)
- Parses tweet JSON from the `--tweet` argument
- Calls `call_minimax(model="MiniMax-M2.5-highspeed", ...)` from common.py with temperature=0.1 (deterministic scoring, but NOT 0.0 — MiniMax rejects 0.0)
- System prompt instructs the model to return JSON with exactly: `relevance_score` (0-100), `category` (one of: freelance_gig, contract_role, remote_job, consulting_lead, automation_prospect, vague_inquiry, not_relevant), `reason` (brief explanation), `should_reply` (boolean), `opportunity_summary` (one line)
- Parses the model response, validates the JSON structure
- On success, stores the scored tweet in the `scored_tweets` table via direct SQLite insert (using all metadata fields from the tweet + score data)
- Outputs: the score JSON `{"relevance_score": N, "category": "...", "reason": "...", "should_reply": true/false, "opportunity_summary": "..."}`
- Logs action with tokens_used and cost_usd extracted from MiniMax response `usage` field

**reply_generator.py** — MiniMax M2.5 full model reply drafting:
- Args: `--tweet` (str, required — JSON string), `--score-data` (str, required — JSON string of scoring output)
- Calls `call_minimax(model="MiniMax-M2.5", ...)` from common.py with temperature=0.7
- System prompt: persona-configurable (reads from a `persona.json` file in scripts dir if exists, falls back to default persona: "AI engineer with 4+ years shipping production ML systems"). Includes rules: under 280 chars, no hashtags, no emoji spam, reference something specific from the original tweet, avoid generic phrases like "I'd love to connect"
- Outputs: `{"reply_text": "...", "char_count": N, "references_original": true/false}`
- Logs action with tokens and cost

**twitter_post.py** — Twikit post/reply wrapper:
- Args: `--action` (str: "reply" or "tweet"), `--tweet-id` (str, optional — required for reply), `--text` (str, required)
- First: calls `check_rate_limit()` from rate_limiter.py. If not allowed, output error with code "RATE_LIMIT_EXCEEDED" and retryable=True
- Then: uses async Twikit client (same cookie pattern as twitter_search.py) to post
- For "reply": calls `client.create_tweet(text=text, reply_to=tweet_id)`
- For "tweet": calls `client.create_tweet(text=text)`
- Outputs: `{"posted": true, "action": action, "tweet_id": response_tweet_id}`
- Logs action to agent_actions table

**report_generator.py** — Lead digest generation:
- Args: `--type` (str: "daily" or "weekly"), `--days` (int, default 1 for daily, 7 for weekly)
- Queries `scored_tweets` table for tweets with `found_at > (now - days*86400)` and `score >= 70`
- Groups by category, sorts by score descending within each group
- For daily: generates a text summary with sections per category, each tweet showing: score, username, opportunity_summary, tweet link (https://twitter.com/{username}/status/{tweet_id})
- For weekly: adds aggregate stats (total found, by category, average score, top 5 leads)
- Outputs: `{"report_type": type, "period_days": days, "total_leads": N, "report_text": "..."}`
- Logs action
